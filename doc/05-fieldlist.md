# Fieldlist ‚Äî Controlling the Perspective

The fieldlist controls how a VirtualTable's columns become rows, columns, and aggregates
in a pivot table. It is the bridge between "what data exists" and "how it is displayed".
Implementation: `Table::Lazy::Fieldlist(T,U)` in `src/fieldlist.cr`.

## Structure

The fieldlist is itself a lazy table (`Raw::Base(T)`) with one row per VT column
and these columns:

```crystal
enum ColumnIndices
    Rank                  # Int64 ‚Äî auto-generated line number (via Raw::Indexed)
    Column                # Int64 ‚Äî which VT column this row describes
    Class                 # Int64 ‚Äî RowClass enum value (see below)
    Level                 # Int64 ‚Äî nesting depth for hierarchical grouping
    SortAscending         # Bool  ‚Äî sort direction for this field
    Name                  # String ‚Äî field name (derived from VT, read-only)
    InternalColumnIndex   # Int64 ‚Äî internal mapping (derived, read-only)
end
```

`Rank` is auto-generated by wrapping the memory table in `Raw::Indexed`.
`Name` and `InternalColumnIndex` are computed by a `Raw::Derived` wrapper ‚Äî
they are read-only projections from the parent VirtualTable.

The underlying storage is a `Raw::Memory(T)` (`@table_memory`) holding only the
first 5 columns (Rank through SortAscending). The full table is `@table`, which
adds the derived columns.

## Classes ‚Äî Row, Column, Aggregate, Unused

Each VT column is assigned to one of four classes, defined in
`Table::Lazy::Pivot::Classes` (`src/table/pivot.cr`):

| Class | Meaning | Example |
|-------|---------|---------|
| `Row` | Groups records into row clusters | Person, City |
| `Column` | Groups records into column clusters | Project, Time |
| `Aggregate` | Displayed in intersection cells | Allocation |
| `Unused` | Hidden from the pivot view | internal IDs |

The user assigns classes by dragging fields between regions in the fieldlist UI
(implemented in `src/gui/fieldlist.cr` and wired via `FieldlistAdapter` in `src/gui/shape.cr`).

## Levels ‚Äî Hierarchical Grouping

The `Level` column (Int64, starting from 0) controls nesting depth.
Multiple levels create hierarchical drill-down in the pivot:

```
 Level 0 Row: Country     ‚Üí outermost row grouping
 Level 1 Row: City        ‚Üí sub-groups within each country
 Level 0 Col: Project     ‚Üí outermost column grouping
 Aggregate:   Allocation  ‚Üí values in intersection cells
```

See [07-pivot-hierarchic](07-pivot-hierarchic.md) for how levels drive nested pivots.

## Sort Direction

`SortAscending` (Bool) controls whether cluster values are sorted ascending or
descending for each field. This affects the ordering of header cells in the pivot.

## Operations

### Normalize

`normalize!` auto-classifies all fields: rank-like fields become `Row`,
everything else becomes `Aggregate`. All levels are set to 0, sort ascending.

```crystal
def normalize!
    # uses @parent.hyperplane_is_rank to detect rank fields
    # rank ‚Üí Row, non-rank ‚Üí Aggregate
end
```

### Empty

`empty!` resets all fields to `Unused` at level 0.

### Mirror Operations

Mirror operations swap field classifications symmetrically:

| Operation | Swaps | Method |
|-----------|-------|--------|
| Horizontal | `Column` ‚Üî `Unused` | `mirror_horizontal_header!` |
| Vertical | `Row` ‚Üî `Unused` | `mirror_vertical_header!` |
| Diagonal | `Row` ‚Üî `Column` | `mirror_diagonal_header!` |
| Horizontal + Agg | Also swaps `Aggregate` ‚Üî `Unused` | `mirror_horizontal!` |
| Vertical + Agg | Also swaps `Aggregate` ‚Üî `Unused` | `mirror_vertical!` |
| Diagonal + Agg | `Row` ‚Üî `Column` + toggle aggregates | `mirror_diagonal!` |

Additionally, the non-aggregate horizontal/vertical mirrors toggle `SortAscending`
on the affected fields.

## Internal Implementation

The fieldlist uses a three-layer table chain:

```
Raw::Memory(T)          actual storage (Rank through SortAscending)
  Raw::Indexed(T)       auto-generates Rank column
    Raw::Derived(T)     adds Name and InternalColumnIndex from VT
```

The `create_table` method builds this chain. Updates are lazy: `update` checks
`@parent.version` and rebuilds only when the VT structure changes (columns added/removed).

When the VT adds or removes columns, the fieldlist adjusts:
- New columns get `Class = Unused`, `Level = 0`, `SortAscending = true`
- Removed columns are deleted from the memory table
- Existing columns preserve their classification

## GUI Adapter

`FieldlistAdapter` in `src/gui/shape.cr` bridges the fieldlist table to the GUI:
- `cell_read(index)` reads fieldlist values
- `cell_assign(index, value)` writes classification changes
- `version` tracks changes for cache invalidation
- `size` returns `{rows, columns}` for the fieldlist grid

The GUI organizes the fieldlist into a nested `RecursiveGrid` layout with drag-drop
zones for each class (Row, Column, Aggregate, Unused), one set per level.
